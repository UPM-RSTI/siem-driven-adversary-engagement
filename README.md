# SIEM-driven Adversary Engagement

**Platform for executing and validating a SIEM alert-driven adversary engagement model based on MITRE Engage.**

This repository accompanies the ARES article:

**Explainable and Interoperable SIEM-Driven Adversary Engagement Based on MITRE Engage**

The main objective is to validate that an enriched SIEM alert can be transformed into a traceable, explainable, and executable MITRE Engage recommendation within a realistic workflow.

---

## Overview

The platform validates the workflow proposed in the Master's Thesis:

1. Load an enriched SIEM alert.
2. Translate the native alert into a normalized alert.
3. Execute the MITRE Engage-based decision model.
4. Generate results containing recommended and non-recommended Activities.
5. Preserve operational information in `payload_context`.

The main pipeline consists of two models:

* **Alert Translator**  
  Translates a native SIEM alert into a common format used by the model.

* **Engage Mapper**  
  Maps the normalized alert to MITRE Engage Goals, Approaches, and Activities while applying the defined exposure thresholds.

### Main Directories

* **`src/`**: contains the frontend of the validation platform. It allows users to load models, configure executions, run the pipeline, and visualize the results generated by the decision model.

* **`app/`**: contains the backend of the validation platform. It also includes the SIEM alert translation model, the MITRE Engage decision model, and the server responsible for exposing the functionality through an API.

* **`MITRE/`**: contains the JSON files used by the decision model. These files represent the required MITRE Engage information, including Goals, Approaches, Activities, their relationships, mappings to MITRE ATT&CK, and CVSS-derived exposure values.

* **`Validation/`**: contains the sample files used to validate the system's operation. It includes an enriched Wazuh SIEM alert and the configuration parameters for the translation model.

* **`public/`**: contains static resources used by the application interface.

## Project Installation

Clone the repository:

```bash
git clone https://github.com/danielMarGar33/siem-driven-adversary-engagement.git
cd siem-driven-adversary-engagement
```

Install the frontend dependencies:

```bash
npm install
```

Install the model backend dependencies:

```bash
pip install fastapi uvicorn
```

---

## Running the Frontend

To build and launch the application in development mode:

```bash
npm run dev
```

Once started, the application will display the local URL for accessing the web interface in the console.

---

## Running the Model Backend

In addition to the frontend, the backend responsible for executing the pipeline models must also be started.  
This backend exposes the endpoints used by the platform to call the **Alert Translator** and **Engage Mapper** models.

To start it locally, run:

```bash
python -m uvicorn webhook_server:app --host 127.0.0.1 --port 5000 --reload
```

Once started, the following hooks will be available:

```text
POST http://localhost:5000/translator
POST http://localhost:5000/engage-mapper
```

These endpoints are used by the platform to execute the models defined in the pipeline:

* `POST /translator`: translates the native SIEM alert into a normalized alert.
* `POST /engage-mapper`: generates the MITRE Engage-based output from the normalized alert.

---

## Basic Usage

The general platform workflow consists of:

1. Start the model backend.
2. Start the frontend.
3. Select the adversary engagement pipeline.
4. Create a new execution.
5. Load a SIEM alert.
6. Configure the fields required to translate the alert.
7. Execute the pipeline.
8. Review the normalized alert, logs, and generated results.

The final output provides the recommended and non-recommended MITRE Engage Activities, together with their exposure values and the rationale behind each decision.

---

## Workflow Structure

The implemented workflow follows this logic:

```text
SIEM Alert
   ↓
Alert Translator
   ↓
Normalized Alert
   ↓
Engage Mapper
   ↓
MITRE Engage Playbook-like Output
```

The normalized alert contains the fields required by the model to make a decision:

* `name`
* `ttps`
* `alert_type`
* `max_cvss_base_score`
* `max_impact_subscore`
* `max_exploitability_subscore`
* `payload_context`

## Docker Deployment

The project can also be run with Docker when deployment as a microservice is required.

Build the image:

```bash
docker build -t siem-engage-decision-engine .
```

Run the container:

```bash
docker run -p 5173:5173 siem-engage-decision-engine
```

If the application's internal port is changed, or if the backend is also deployed in a container, adjust the port mapping accordingly.
